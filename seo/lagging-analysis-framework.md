# 成效落後分析框架 · 早安毛健康

> 用途：分析 SEO 文章成效落後或警示狀況時，依此框架排查、量化、做 ROI 取捨。
> 來源：Welly 內部《成效落後分析【知識庫】》案例庫與分析問題庫的精煉版。
> 與其他 reference 分工：本檔聚焦「成效落後的判斷與排序方法」；個別問題的修法落到 `eeat-checklist.md`、`google-algo-signals.md`、`brand-voice.md`、`revision-types.md`。

## 一、分析的 6 大維度（必排查）

每篇成效落後文章先用以下維度逐項排查，不要憑直覺跳到結論。

| 維度 | 主要問題類型 | 對應 reference |
|------|-------------|---------------|
| 一、搜尋意圖 | 與 SERP 不符、混雜、最近發生變化 | 本檔 § 二 |
| 二、文章類型 | 與 SERP 主流類型不符、需拆分／合併 | 本檔 § 二 |
| 三、Content SEO | Meta、H 標籤、內容深度、E-E-A-T、AI 感、scaled content | eeat / brand-voice / google-algo |
| 四、Technical SEO | head 重複、canonical、麵包屑、PageSpeed、5xx、索引狀態 | 本檔 § 三 |
| 五、競品條件 | DR、EEAT 級別、文章量、產業類型 | 本檔 § 五 |
| 六、整站體質 | Helpful Content 站點分類、unhelpful 比例、站內競爭 | google-algo § 一.1 |

排查完畢用 § 六 的格式產出「推測主要原因 → 對應調整 → 量化指標」三段式結論。

## 二、搜尋意圖與文章類型一致性（量化標準）

> 知識庫核心兩條判準：

| 判準 | 一致性標準 |
|------|-----------|
| 搜尋意圖與 SERP 重複率 | ≥ **40%** 視為一致 |
| 文章類型與 SERP 重複率 | ≥ **80%** 視為一致 |

### 怎麼算重複率

取 Google SERP 前 10 名 organic results：
- **搜尋意圖**：每一筆判讀「主導意圖」（informational / transactional / navigational / commercial）；自家文章意圖落在多數派裡 → 重複；少數派 → 不重複
- **文章類型**：每一筆判讀類型（綜合指南 / 列表推薦 / 純價格表 / 商品頁 / 比較文 / 詞條百科 / 影片）；自家文章類型落在多數派裡 → 重複

### 混雜情境的處理

**SERP 混雜（混合多種類型）** → 以「該文章所選目標關鍵字整體」做判斷：
- 如果整組目標字（核心 + 長尾）都跟 SERP 多數派一致 → 維持本頁優化
- 如果整組目標字多數與 SERP 不符 → 評估換追蹤頁 / 拆篇 / 合併（參考案例庫【倉儲設備】、【聯誼推薦】、【品牌 logo】）

**判斷沒救的情境**：
- 例 1：SERP 9 成是電商商品／價目表，客戶 DR 與相關性都做不過，且優化內容或寫小文都無效 → 確認整體進程是否落後，若僅這個字落後，可先放著（參考案例【燕窩價格】）
- 例 2：搜尋意圖混雜，且自家無對應頁面更適合 → 嘗試把 SERP 出現的子題都納入文章，但每個子題篇幅不用太長（參考案例【送禮】）

### 取得 SERP 資料的工具

**步驟 2 分析階段 → 一律用 SerpAPI（`fetch_serp.py`）：**

```bash
python3 "{SKILL_DIR}/scripts/fetch_serp.py" --keyword "{關鍵字}" --out "{RUN_DIR}/01-serp.json"
```

- 回傳結構化 JSON（organic_results / related_searches / people_also_ask）
- 每個目標字至少跑一次，不要憑印象判斷
- 失敗時停下告訴使用者，不自行降級到 WebSearch

**步驟 5 改文階段（事實查證 / URL 驗證）→ 用 WebSearch 或 Chrome 瀏覽器工具。**

不再使用 Ahrefs Keywords Explorer 或無痕視窗取 SERP 排名資料。

## 三、Technical SEO 排查與優先級分類

> 這層的取捨原則：**「真技術 bug」與「展示加分項」要分開**。
> Schema 只影響 SERP 展示效果（星級、卡片），大多數情況不直接影響排名 → 屬展示加分項；但有些技術問題會**直接影響索引／可訪問性**，那是真 bug，必修。

### 3.1 真技術 bug（高優先，直接影響索引／排名／流量）

以下任何一條出現 → 必標為「優先修改」並推工程處理：

| 問題 | 影響 |
|------|------|
| `<head>` 區出現重複 `<title>` / `<meta description>` / `<link canonical>` | 多個 canonical 互相打架，Google 可能選錯版本，最糟把該頁視為他頁副本而從索引移除；多個 title 在不同爬蟲下選擇不一致 |
| canonical 指向錯誤 URL（指到首頁、指到分頁、指到下架頁） | 等同告訴 Google「這頁不是 main version」→ 不索引或合併索引 |
| 大量 5xx 錯誤頁面 | 整站可被視為不穩定，影響爬蟲頻次（參考案例【艾希居】） |
| 同一頁有 www 與無 www / 帶斜線與無斜線 / http 與 https 兩版本同時可訪問 | 整站重複內容（參考案例【好眠呼吸醫材】） |
| 帶 UTM／參數的網址被收錄為獨立索引 | 重複內容、稀釋權重（參考案例【OK 忠訓】） |
| 網站被惡意建立子頁面且被索引 | 站點層級信任傷害（參考案例【米青品牙醫】） |
| 重要頁面被 robots / noindex 擋 | 整篇不會出現在 SERP |
| 目標頁狀態碼異常（GSC 200、Ahrefs 404 不一致） | 工程排查爬蟲端是否有渲染問題（參考案例【戀愛大學】） |

### 3.2 展示加分項（低-中優先，影響 CTR 不影響排名）

以下屬於「好還要更好」類別。**只在「客戶基礎已扎實」或「主要競品都做了所以你不做會吃虧」兩種情境下才動**：

| 問題 | 影響 |
|------|------|
| Schema / 結構化資料（Article、FAQPage、HowTo、BreadcrumbList） | 影響 Rich Result 卡片，提升 CTR；不影響排名 |
| Open Graph / Twitter Card | 影響社群分享預覽；不影響排名 |
| 圖片 alt（少量缺漏） | 影響圖片搜尋；對排名影響微小（知識庫範例 1：1 張缺 alt 可不修） |
| H 標籤微調（個別空白 H3） | 影響閱讀體驗；對排名影響微小（知識庫範例 2：2 處空白 H3 可不修） |

> **原則**：基礎沒做好（內容缺失、E-E-A-T 弱、有真技術 bug）時，不應該先動 Schema 或 Twitter Card。把資源放在更有 ROI 的地方。

### 3.3 PageSpeed 處理優先級

依案例庫【歪國零食嘴】【囍聚手工婚紗】【增好貸】整理：

| 項目 | 處理方式 |
|------|---------|
| 圖片過大 | **優先處理**：可請編輯換 JPG / WebP 並壓縮；WordPress 站可用 WP Rocket 處理畫面外載入 |
| 文字壓縮（Enable text compression） | 影響速度超過 10 秒再留意，提需求給 PM |
| 減少無用 JS / CSS | 影響速度超過 20 秒再留意；通常無解，需架站時就注意 |
| 行動裝置分數 60 以上 | **算及格**，60 以下才優先排查 |

## 四、ROI 思考（決定要不要修這個問題）

排查到問題不一定要每個都修。每個問題都用以下 4 題自問：

1. **這個問題是否直接影響排名或網站流量？**
   - 是 → 進入評估
   - 不是（純展示／純 CTR 微調） → 降級處理或不處理
2. **修這個問題需要多少資源（時間、人力、溝通成本）？**
   - 工程資源：要不要動代碼／要不要溝通客戶工程
   - 編輯資源：要不要重寫多少段
   - PM 溝通：要不要說服客戶
3. **修完之後能否量化結果？**
   - 能（排名、流量、CTR、索引狀態） → 值得追蹤
   - 不能（純體感／無從驗證） → 降級
4. **有沒有更省力達到同個目標的方式？**
   - 有 → 換方式（例：站內競爭可以用 301 + 內容合併，不一定要改追蹤 URL）
   - 沒有 → 老老實實做

### ROI 取捨範例

| 問題 | 錯誤判斷 | 正確判斷 |
|------|---------|---------|
| 內文有 1 張照片 alt 沒設置 | 必修 | 不修（影響微小，工時可投在更有 ROI 的事） |
| 文章有 2 處空白 H3 | 必修 | 不修（影響微小） |
| Schema 缺 FAQPage | 必修 | **看情境**：客戶基礎已扎實且競品都有 → 補；基礎沒做好（內容缺失、E-E-A-T 弱）→ 不修 |
| `<head>` 重複 canonical 指錯首頁 | 不重要（看起來小問題） | **必修**（直接影響索引） |
| 整站有 4 個惡意子頁被索引 | 慢慢處理 | **必修**（站點信任，先 GSC 移除） |

## 五、競品條件評估

落後分析時要看「自己 vs 競品」的相對條件：

| 維度 | 怎麼看 | 對應行動 |
|------|-------|---------|
| 競品 DR | Ahrefs Domain Rating | 若客戶 DR 顯著低於競品 → 補外鏈、Web2.0 |
| 競品 EEAT | 是不是 .edu / .org / 醫院 / 政府 / 國際品牌？是否有醫師掛名？ | YMYL 產業競品 EEAT 高 → 補來源、補研究、補機構引用、加掛名（參考案例【膝蓋痛】【法令紋】【prp】） |
| 競品文章量 | 站上同主題文章數 | 競品文章量大 → 安排相關性小文（參考案例【產後掉髮】【冷凍減脂】） |
| 競品產業類型 | 是電商部落格、媒體、診所、品牌、論壇？ | 自家是純媒體、競品是電商 → 用內容深度 + 第一手經驗突圍（不要靠商品線打） |
| 競品內容架構 | 看 SERP 前 3 名章節分布 | 自家文章類型若與多數一致 → 沿用主流架構；不一致 → 看 ROI 決定要不要重寫 |

**特殊情境**：
- 自家網站體質弱（部落格架構、易被演算法攻擊）→ 改寫文章為更個人化視角，降低 SEO 感（參考案例【竹北美食】）
- 客戶不能掛醫師名 → 用「引用研究 + 國外權威來源」+ 站上有醫師專欄文章的醫師話 + 醫師署名（參考案例【捷元生技】）

## 六、結論的格式

排查完畢的結論用以下三段式產出（對應知識庫案例庫格式）：

```markdown
### 推測主要原因
（占比由大到小列 2-4 條，每條一行，標清楚是觀察到的事實還是推測）
- 內容缺失（佔 60%）：H2「怎麼養」與「好養嗎」同議題寫兩次（觀察）；無第一手經驗痕跡（觀察）
- E-E-A-T 不足（佔 30%）：HCM/SMA/PKD 零國外權威引用，YMYL 紅線（觀察）
- Technical 風險（佔 10%）：head 區重複 canonical 指向首頁（觀察，但實際影響需 GSC 驗證）

### 對應調整
（依優先級排：架構先、E-E-A-T 次、表達層後、Technical 平行）
1. 架構：合併同議題 H2、補 1 段第一手經驗
2. E-E-A-T：3 個遺傳病補 Cornell / VCA / UC Davis 來源
3. 表達：個性段去 4 排比小標、結尾去客套
4. Technical：請工程修 SEO component 重複標籤問題

### 量化指標（後續驗證用）
- 大改完成 → 4 週 / 8 週 / 12 週各取 GSC 數據對比
- 4 週進前 20、8 週進前 10 視為大改有效（參考案例【車貸繳不出來】時程）
- 同步觀察 Ahrefs Organic keywords 數量、Ahrefs DR
```

## 七、客觀性自查（避免主觀臆斷）

落後分析時最常見的錯誤是「憑經驗下結論」。寫完結論前自查：

1. **每個推測都標清楚是「觀察事實」還是「需驗證假設」？**（不可把推測寫成事實）
2. **量化都有來源嗎？**（佔比 60% / 競品 6/8 是電商 / DR 差距 N — 都要可追溯）
3. **有沒有拿不同網站的狀況直接類比？**（YMYL vs 商業、寵物 vs 醫美，行業差異要交代）
4. **如果結論是「沒救」（參考案例【冷凍減脂】），是否真的逐項排查過？**還是太快放棄？
5. **SerpAPI / GSC / Ahrefs 三類資料至少看了一個嗎？**（沒有就不該下「主要原因」結論）

任一題答「否」 → 補上資料或改寫結論為「待驗證假設」。
