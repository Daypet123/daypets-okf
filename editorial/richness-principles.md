# 豐富性原則（Richness Principles）

> 用途：02-structure 規劃 H2／H3 時的段落覆蓋判準；03-writer 撰文的內容密度下限；04-qa-polish 最終審核的豐富度檢核。
> 本檔新增自 2026-04-17 第一篇驗證後的改善。

---

## 一、段落覆蓋 = Parity + Gap-Fill

Welly 的文章架構不走「原創腦補」，也不走「照抄競品」，而是：

```
段落覆蓋 = 競品交集（Parity，必須有）+ 使用者旅程缺口（Gap-Fill，差異化補）
```

### 1.1 Parity：競品有的段落資訊，我們也要有

- SERP 前 3 名競品都寫的 H2 主題 = SERP 共識，我方文章必須涵蓋
- **但不得抄襲**：必須換角度、換句序、換實例
  - 換角度：例如競品用「病理機轉」說明疾病，我方改用「飼主在家觀察到的行為異常」切入
  - 換句序：例如競品「原因 → 症狀 → 治療」，我方改「症狀 → 自我判斷 → 原因 → 治療」
  - 換實例：例如競品引用歐美數據，我方換台灣本地案例或具體店家、平台、政府網站
- 02-structure 規劃此類 H2 時必須標注：「Parity（對應 SERP #N 的 H2 名稱）」＋「換角度策略：____」

### 1.2 Gap-Fill：競品沒寫但用戶旅程上重要的主題

- 來源：
  - Google 本頁的「People also ask」「related searches」
  - 客戶 `client-knowledge.md` 裡記錄的常見客訴或 FAQ
  - 品牌差異化角度（客戶獨家 know-how、客戶服務對象的特殊需求）
  - 商業漏斗下游問題（競品寫「是什麼」，我方補「買了之後怎麼做」）
- 02-structure 規劃此類 H2 時必須標注：「Gap-Fill（差異化理由：____）」
- **Gap-Fill H2 數量上限**：不超過總 H2 數的 40%。避免偏離搜尋意圖、傷害 SEO 表現。

### 1.3 細部清單同理（Parity 內部）

每個 H2 段落裡的內部 list（疾病清單、顏色清單、FAQ 題數、花色清單等）也適用 Parity：

- SERP 前 3 名列了 6 種常見疾病，我方若只寫 4 種 → 02-structure Gate Check 會 fail
- 例外：若客戶品牌立場刻意精簡（例：「我們只挑最重要的 3 種」），必須在 02-structure 撰寫方向裡註明理由，否則視為缺漏

## 二、具體量化數字下限

任何涉及「該多久一次／該多少量／該多少錢」的段落，必須給具體數字或區間：

| ✅ 正確寫法 | ❌ 常見抽象寫法 |
|---|---|
| 每週梳毛 2–3 次，換毛季增至每日一次 | 定期梳毛即可 |
| 成貓每日 3–4 餐、定時定量 | 定時餵食、避免暴食 |
| 每 1–2 個月洗澡一次 | 定期洗澡 |
| 寵物級 2.5–7 萬、繁育級 4–15 萬、賽級 7–30 萬 | 價格 3 萬到數十萬不等 |
| 每日互動遊戲 15–30 分鐘 | 每天陪牠玩 |

### 價格段落的分級規則

凡是「XX 價格」「XX 多少錢」這類 H2，**不得只給單一數字帶過**。必須給至少 3 個分級或場景：

- 血統／等級分級（寵物級 / 血統級 / 賽級）
- 通路分級（品牌直營 / 專賣店 / 網購）
- 產品定位分級（入門 / 標準 / 旗艦）

三種都不適用的主題（極少），要在撰寫方向註明「本主題無法分級」的理由。

## 三、具名資源 > 抽象描述

外部資源（領養平台、政府網站、社群、認證機構、醫療機構等）必須用**真實名稱**並帶超連結，不得以通用描述含混。

| ✅ 正確寫法 | ❌ 常見含混寫法 |
|---|---|
| 「MeetPets 台灣認養地圖」 | 「民間送養平台」 |
| 「全國動物收容系統」 | 「各地公立收容所」 |
| 「TICA、CFA 國際貓協會認證血統書」 | 「國際血統認證」 |
| 「政府核發的特定寵物業許可證（俗稱特寵業繁字號）」 | 「合法貓舍許可」 |
| 「Meta（Facebook）社團『布偶貓領養』」 | 「某些 Facebook 社團」 |

**分工**：

- 01-research 要抓競品提到的所有具名資源（平台、機構、認證、產品），整理進 `01-research.md` 的「具名資源候選」段落
- 02-structure 在撰寫方向註明哪些段落「必須使用具名資源」
- 03-writer 引用時必嵌超連結（用戶無法點擊的具名資源不算數）
- 04-qa-polish 掃抽象詞彙（「某平台」「一般來說」「通常」）並改寫

## 四、涵蓋量的驗證時機

| 階段 | 檢查動作 |
|---|---|
| 01-research | 記錄 SERP 前 3 名的 H2 列表 + 其中內部清單的量體（疾病 N 種／FAQ M 題／花色 K 種） |
| 02-structure | H2 對 Parity 做 1:1 對照；若缺主題、或內部清單量少於中位數 → 由結構師補上並註明換角度策略 |
| 02-structure Gate | Parity 未覆蓋即 fail，**硬規則退回 02-structure 重跑** |
| 03-writer | 按 02-structure 指示的覆蓋量撰寫（不得偷懶減項） |
| 04-qa-polish | 最後掃一遍：有沒有抽象資源、有沒有無分級的價格段、有沒有「定期／適量」這類空話 |

---

## 五、豐富性不是湊字數

此原則的目的是「覆蓋使用者實際需求」，不是把文章灌水：

- Parity 是覆蓋 SERP 共識，不是多寫就好；若 SERP 前 3 共識只有 5 個 H2，我方就 5 個 H2，不硬加第 6 個
- Gap-Fill 要有差異化理由；沒理由就別加
- 具體數字與具名資源的引入必須來自 01-research 抓到的實際來源，**不得虛構**（憑空編出「TICA/CFA」之類的機構名稱是嚴重違規）
- 寫到某段找不到真實資源或具體數字 → `qa-flags.md` 請使用者補資料，不得編造
