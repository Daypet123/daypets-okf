# 文章與架構的寫作品質標準

> 來源：`文章與架構的寫作品質標準.docx`
> 用途：02-structure 規劃每段撰寫方向時引用；03-writer 撰文時遵循；04-qa-polish 審稿判準。

## 一、段落結構：魚骨寫作法

每個段落都像一條魚：

- **魚頭（主題句）**：第一句就給結論或核心觀點，讀者掃讀時只看這句就能抓到重點
- **魚骨＋魚肉（補充）**：用細節、數據、案例支撐主題句，讓文章生動有說服力
- **魚尾（收尾）**：重複強化核心觀點，加深讀者印象

在 `02-structure.md` 的應用：每個 H2 的「撰文方向」標注該段的「魚頭」是什麼（核心結論），讓撰文者開頭就破題。

---

## 二、文章結構：金字塔法

- 先講結論，再用補充強化結論
- 讀者注意力有限，鋪陳太久會變流水帳
- H2 若有過渡段，則是該章節的「金字塔頂端」；H2 沒過渡段時，第一個 H3 或第一段 bullet 承擔頂端結論（2026-04-20 A5 裁決：過渡段非強制）

---

## 三、六大技巧（撰文方向標注用）

在 `02-structure.md` 中，每個 H2 的「撰文方向」建議標注 1-2 個適合的技巧代號。

| 代號 | 技巧 | 適用情境 | 範例 |
|:---:|------|---------|------|
| T1 | 用已知比較未知 | 讀者對新概念陌生時 | 先講「養貓」的經驗，再對比「養兔」的差異 |
| T2 | 用具體比喻抽象 | 解釋抽象概念時 | 「SEO 就像虛擬世界的黃金地段」 |
| T3 | 點出讀者相關性 | 需要讓讀者覺得「跟我有關」 | 附上台灣的具體數字或統計 |
| T4 | 連結生活經驗 | 讓知識與日常行為連結 | 「你每天滑 IG 看到的推薦，背後就是這套機制」 |
| T5 | 打臉讀者 | 製造認知衝突引發好奇 | 「你以為 XX 是 YY？其實完全相反。」 |
| T6 | 破題解答困惑 | 回應讀者長久的疑問 | 「常常聽說 XX 但搞不懂？這篇一次講清楚。」 |

## 四、文章好壞的判斷標準

> 文章好壞取決於是否符合寫作目的。

在架構階段就要明確：

1. 這篇文章的「寫作目的」是什麼（教育、比較、導購、解惑）
2. 目標讀者讀完後應該「帶走什麼」（一個結論、一組行動方案、一個購買決定）
3. 每個 H2 對達成這個目的有什麼貢獻

如果某個 H2 無法清楚說明它對文章目的的貢獻，就該刪掉或合併。

## 五、事實查證（Fact-Check）

> 適用所有客戶。QA 階段（Phase B）必須執行，不可省略。

### 查證原則

1. **來源歸因必須精確**：文章中寫「根據 X 機構」，X 機構的原始頁面就必須包含該數據。張冠李戴（把 A 來源的數據歸到 B 來源）是最常見的錯誤類型
2. **用 web search 驗證，不靠內部知識**：每一筆具體數據（死亡率、潛伏期、劑量、濃度、年齡範圍等）都必須用 web search 查原始來源比對，不可僅靠 AI 訓練資料
3. **URL 必須存活**：所有文內超連結必須實際可訪問（非 404），用 web_fetch 驗證。若 URL 失效，尋找替代來源或移除該引用
4. **避免過度絕對**：「只」「一定」「絕對」等用詞需審慎，確認來源是否支持如此絕對的表述。若來源用「主要」「通常」等限定詞，文章也應保留
5. **絕對禁止捏造來源內容（零容忍）**：如果無法實際讀取某個網頁的內容（被爬蟲擋住、403、需要登入、內容為空等），**不可以假裝已經讀過並編造該來源說了什麼**。這比不放連結嚴重一百倍——等同欺騙讀者和客戶。寧可整段不引用社群素材，也不準捏造引述內容
6. **能收集到什麼就說什麼**：不要為了讓文章看起來更豐富、更有「真人案例」而編造具體數據和引述。沒抓到就如實告知，或改用其他可驗證的素材

### 常見錯誤模式（撰文 + QA 時留意）

| 錯誤模式 | 說明 | 範例 |
|----------|------|------|
| 張冠李戴 | 數據正確但歸因錯誤 | 死亡率 90% 來自 ABCD，卻寫成「根據 Cornell」 |
| 過期數據 | 引用舊標準，新版已修正 | 漂白水濃度從 5.25% 更新為 5～6% |
| 過度絕對 | 來源有限定詞但文章省略 | 來源寫「主要感染貓科」，文章寫「只感染貓科」 |
| 範圍錯誤 | 數字區間與文獻不符 | 文獻寫 2～7 天，文章寫 2～10 天 |
| 療效誇大 | 將「實證有限」寫成「可縮短病程」 | 干擾素臨床試驗未證實，但文章語氣肯定 |
| 只取上限 | 僅引用極端值忽略多數情況 | 排毒「最長 6 週」但多數 2 週內停止 |
| **捏造引述** | 無法讀取來源內容卻編造具體細節掛上連結 | Dcard 原文只是提問，卻捏造「住院費每天 2,500 元、干擾素 7,500 元」等帳單明細歸因給該網友。**這是最嚴重的錯誤，零容忍** |

### 社群來源（Dcard / PTT / 論壇）抓取規則

社群平台經常有反爬蟲機制，導致 web_fetch 拿到空白或 403。處理原則：

1. **抓不到內容 → 不准引用**：如果 web_fetch 和瀏覽器都讀不到某篇社群貼文的實際內容，就不可以在文章中引述該貼文的任何具體細節。連結也不要放
2. **替代平台優先順序**：Dcard / PTT 抓不到時，改搜尋 Reddit、Threads、Facebook 公開貼文、部落格等可實際讀取內容的平台。有真實素材才寫，沒有就不寫
3. **已抓到但內容不符 → 立即移除**：如果實際讀到的內容跟文章引述的不一致，該段引述必須立即刪除或改寫為與實際內容相符的版本
4. **社群素材是加分項，不是必要項**：文章品質不靠捏造的網友案例撐場。沒有合適的社群素材，用權威來源（獸醫學會、教科書）寫出的段落一樣好

### 查證流程

1. **逐條掃描**：列出文章中所有含具體數據、來源歸因、絕對性表述的句子
2. **web search 比對**：針對每條，搜尋原始來源文本，確認數據與歸因一致
3. **社群引述驗證**：所有引用社群平台（Dcard、PTT、論壇）的段落，必須實際讀取原文確認內容吻合。讀不到 → 移除引述或換來源
4. **URL 驗證**：用 web_fetch 測試所有文內超連結是否正常回應
5. **產出查證報告**（選用）：若錯誤 ≥ 3 項，產出 `{RUN_DIR}/fact-check-report.md` 記錄每項錯誤的原文、事實、來源 URL、修正方式
6. **修正後重跑驗證 script**：確保修正沒有引入新的格式問題
